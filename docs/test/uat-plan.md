# User Acceptance Testing (UAT) Plan

Meeting Minutes Automator - Google Docs同期機能のユーザー受け入れテスト計画。

## 概要

| 項目 | 内容 |
|------|------|
| 対象機能 | Google Docs同期 (MVP2) |
| テスト期間 | 1週間 |
| 参加者数 | 3名以上 |
| 成功基準 | クリティカルバグ0件、満足度80%以上 |

---

## 1. テスト参加者

### 参加者プロファイル

| ID | 役割 | 技術レベル | 使用頻度 |
|----|------|------------|----------|
| U1 | プロジェクトマネージャー | 低 | 週5回 |
| U2 | エンジニア | 高 | 週3回 |
| U3 | 営業 | 中 | 週2回 |

### 参加条件

- Google Meetを定期的に使用している
- Chrome ブラウザを使用している
- Googleアカウントを持っている

---

## 2. テスト環境

### 必要なもの

- [ ] Chrome ブラウザ (最新版)
- [ ] Google アカウント
- [ ] Google Docs ドキュメント（テスト用）
- [ ] Tauri アプリ（インストール済み）
- [ ] Chrome 拡張機能（インストール済み）

### 環境セットアップ手順

1. Tauri アプリをインストール
2. Chrome 拡張機能をインストール
3. Google アカウントで認証
4. テスト用 Google Docs ドキュメントを作成

---

## 3. テストシナリオ

### シナリオ A: 初回セットアップ

**目的**: 初めてのユーザーがセットアップできるか確認

| ステップ | 操作 | 期待結果 |
|----------|------|----------|
| A1 | 拡張機能アイコンをクリック | ポップアップが表示される |
| A2 | 「Googleアカウントと連携」をクリック | 認証画面が表示される |
| A3 | Googleアカウントでログイン | 認証成功、設定画面が表示される |
| A4 | ドキュメントIDを入力 | 入力が保存される |
| A5 | 同期を有効化 | ステータスが「同期中」になる |

**所要時間**: 5分

### シナリオ B: 通常の会議フロー

**目的**: 実際の会議での使用感を確認

| ステップ | 操作 | 期待結果 |
|----------|------|----------|
| B1 | Google Meet会議を開始 | 会議が開始される |
| B2 | Tauriアプリで録音開始 | 録音が開始される |
| B3 | 30秒間話す | 文字起こしが表示される |
| B4 | Google Docsを確認 | 文字起こしが反映されている |
| B5 | 追加で話す | リアルタイムで追記される |
| B6 | 録音停止 | 同期が完了する |

**所要時間**: 10分

### シナリオ C: オフライン→オンライン復帰

**目的**: ネットワーク切断時の動作を確認

| ステップ | 操作 | 期待結果 |
|----------|------|----------|
| C1 | 録音中にWiFiをオフ | オフライン表示になる |
| C2 | 10秒間話す | 文字起こしがキューに保存される |
| C3 | WiFiをオン | オンライン表示になる |
| C4 | Google Docsを確認 | キューの内容が同期されている |

**所要時間**: 5分

### シナリオ D: エラーからの復旧

**目的**: エラー発生時のユーザー体験を確認

| ステップ | 操作 | 期待結果 |
|----------|------|----------|
| D1 | 無効なドキュメントIDを入力 | エラーメッセージが表示される |
| D2 | 正しいドキュメントIDに修正 | 同期が再開される |
| D3 | ログアウト | 連携解除される |
| D4 | 再ログイン | 同期が再開される |

**所要時間**: 5分

## 4. フィードバック収集

### アンケート項目

**満足度 (1-5)**:
- [ ] セットアップの容易さ
- [ ] 同期の速度
- [ ] エラーメッセージの分かりやすさ
- [ ] 全体的な使いやすさ
- [ ] 推奨度 (NPS)

**自由回答**:
- 良かった点
- 改善してほしい点
- バグ報告
- 機能リクエスト

### フィードバック収集方法

1. **テスト中**: Slackチャンネルでリアルタイム報告
2. **テスト後**: Googleフォームでアンケート回答
3. **インタビュー**: 希望者に30分の1on1インタビュー

## 8. リスクと対策

| リスク | 影響 | 対策 |
|--------|------|------|
| 参加者不足 | テスト不完全 | 早期リクルーティング |
| 環境構築失敗 | テスト遅延 | 事前セットアップサポート |
| クリティカルバグ発見 | リリース延期 | 即時修正体制 |
